01 The Problem: Why DataFrames Are Slow

Imagine you have a dashboard showing sales totals by region. You have 1 million rows of data. Every time a new sale comes in, you need to update the totals.

With traditional DataFrame libraries (like Pandas in Python or typical JS implementations), adding that one new row means re-reading and re-processing all 1 million rows.

‚ùå Traditional Approach O(n)

// Every update recomputes EVERYTHING
data.push(newSale);
const totals = data.reduce((acc, row) => {
  acc[row.region] = (acc[row.region] || 0) + row.amount;
  return acc;
}, {});
// ‚Üë Scans 1,000,001 rows every time!

‚úì Incremental Approach O(Œî)

// Only process what changed
source.insert([newSale]);
// Internally:
// 1. Look up "West" group ‚Üí O(1)
// 2. Add 150 to sum ‚Üí O(1)
// 3. Done!
// ‚Üë Processes just 1 row

The Core Insight

Most operations (filter, map, groupBy) have a mathematical property: you can apply them to just the changes and combine with previous results. This is called incrementalization.

03 Weighted Rows: The Magic of +1 and -1

Here's where incremental processing gets interesting. Instead of just storing data, we attach a weight to every row:

  • +1 means "this row exists" (insertion)
  • -1 means "remove this row" (deletion)
  • 0 means "row cancelled out" (no effect)

This simple concept is called a Z-Set (or weighted set), and it's the foundation of a technique called DBSP (Database Stream Processing).

Z-Set Example: Tracking Changes
Weight Region Amount Meaning
+1 West 100 Insert: West sale of $100
+1 East 200 Insert: East sale of $200
-1 West 100 Delete: Remove that West sale
+1 West 150 Insert: New West sale of $150

Why Weights Are Powerful

The key insight is that most operations distribute over weights. This means instead of recomputing everything, we can:

  1. Apply the operation to just the delta (new changes)
  2. Combine the result with our previous state
  3. Weights automatically handle insertions AND deletions!
// Mathematical property: filter distributes over addition
filter(A + B) = filter(A) + filter(B)

// So if we have:
//   previousResult = filter(existingData)
//   delta = newChanges (with weights +1 or -1)
// Then:
//   newResult = previousResult + filter(delta)
//                                ‚Üë Only process the delta!

Deletions Are Just Negative Insertions

When you delete a row, you don't scan your data to remove it. Instead, you add a row with weight -1. When it flows through your aggregations, it subtracts from the totals automatically.

Filter Operator (Stateless)

Filter is beautifully simple: it just sets weights to zero for rows that don't match. 

class FilterOperator {
  step(input: ZSet): ZSet {
    // Evaluate predicate for each row
    const mask = this.evaluatePredicate(input);
    
    // Keep weight if passes, zero if fails
    const newWeights = new Int32Array(input.numRows);
    for (let i = 0; i < input.numRows; i++) {
      newWeights[i] = mask[i] ? input.weights[i] : 0;
    }
    
    return input.withWeights(newWeights);
  }
}

GroupBy Operator (Stateful)

GroupBy maintains running aggregates for each group. When a new row arrives, it updates just that group's state: 

class GroupByOperator {
  private groups = new Map();  // key ‚Üí accumulator state
  
  step(input: ZSet): ZSet {
    for (let i = 0; i < input.numRows; i++) {
      const weight = input.weights[i];
      if (weight === 0) continue;
      
      const key = this.computeKey(input, i);
      const group = this.groups.get(key) ?? this.createGroup();
      
      // Update with weight: +1 adds, -1 subtracts!
      group.sum += input.getValue('amount', i) * weight;
      group.count += weight;
      
      this.groups.set(key, group);
    }
    
    return this.computeOutputDelta();
  }
}
üîë

Notice how weight naturally handles both insertions and deletions. When weight = -1, we subtract from the sum and decrement the count. No special deletion logic needed!

05 State Management: How Accumulators Work

The secret sauce of incremental aggregation is the accumulator‚Äîa tiny state machine that knows how to update results incrementally.

Sum Accumulator

The simplest case: just add value √ó weight to the running total.

Mean Accumulator

Mean requires tracking both sum and count. The weight affects both! 

class MeanAccumulator {
  state = { sum: 0, count: 0 };
  
  accumulate(value: number, weight: number) {
    this.state.sum += value * weight;
    this.state.count += weight;  // Count also uses weight!
  }
  
  finalize(): number | null {
    if (this.state.count === 0) return null;
    return this.state.sum / this.state.count;
  }
}

Min/Max: The Tricky Case

Min and Max are harder to incrementalize. Why? Because when you delete the current minimum, you need to find the next smallest value.

Min/Max State Complexity

Unlike sum (O(1) state), min/max must track all distinct values and their counts. When the current min is deleted, scan the state to find the new min. State size: O(distinct values). 

class MinAccumulator {
  // Must track ALL values, not just the minimum
  state = new Map<number, number>();  // value ‚Üí count
  
  accumulate(value: number, weight: number) {
    const currentCount = this.state.get(value) ?? 0;
    const newCount = currentCount + weight;
    
    if (newCount <= 0) {
      this.state.delete(value);  // Value no longer present
    } else {
      this.state.set(value, newCount);
    }
  }
  
  finalize(): number | null {
    if (this.state.size === 0) return null;
    return Math.min(...this.state.keys());
  }
}

Why It Works

Cache-Friendly Memory Access

TypedArrays store data contiguously. When summing a column, the CPU prefetcher loads upcoming values automatically. Row-based access jumps around memory randomly.

No Boxing Overhead

JavaScript numbers are normally wrapped in objects (~16 bytes each). TypedArrays store raw 8-byte values. For 1M numbers: 8 MB vs 16+ MB.

JIT Optimization

V8 generates highly optimized machine code for TypedArray loops. Benchmarks show 4.9√ó speedup after JIT warmup (vs 1.4√ó for row-based).

Work Proportional to Change

Incremental processing does O(Œî) work regardless of total data size. 1 update to 1 million rows = same cost as 1 update to 100 rows.

When to Use This

This approach shines for dashboards, real-time analytics, and streaming data‚Äîanywhere you have large datasets with frequent small updates. The benefits are multiplicative: columnar √ó incremental = massive wins.

Quick Reference: Implementation Recommendations

  1. For numeric-heavy workloads: Use Float64Array / Int32Array. 30-80√ó faster than arrays of objects.
  2. For streaming updates: Implement weighted Z-Sets. Even 10% deltas give 7√ó speedup.
  3. For cross-platform data: Apache Arrow gives ~3.6√ó speedup and zero-copy interop with Python/Spark.
  4. For string columns: Dictionary-encode when <50% unique values. Compare integers instead of strings.
